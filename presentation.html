What is Machine Learning?

Formal Definition (Mitchell, 1997)

A computer program is said to learn from experience E with respect to some task T and some performance measure P, if its performance on T, as measured by P, improves with experience E.

Supervised

Data: Labeled examples (input + desired output).

Goal: Learn a mapping from inputs to outputs.

Core Tasks:

  • Classification: Predict discrete labels (e.g., spam vs. ham).
  • Regression: Predict continuous values (e.g., house prices).

Unsupervised

Data: Unlabeled data (no teacher).

Goal: Discover hidden patterns or structures.

Core Tasks:

  • Clustering: Group similar items.
  • Dim. Reduction: Simplify data without losing info.
  • Anomaly Detection: Flag outliers.

Other Learning Axes

How they generalize:

Instance-based

Memorize examples, then compare new data (e.g., k-NN).

Model-based

Build a model from examples, then predict (e.g., Linear Reg).

Batch vs. Online:

  • Batch: Train on all data at once.
  • Online: Train incrementally as data arrives.

Reinforcement Learning: An agent observes the environment, selects actions, and gets rewards or penalties (e.g., AlphaGo).

What is Machine Learning?

Slide 2 / 15

Arthur Samuel (1959)

"The field of study that gives computers the ability to learn without being explicitly programmed."

Task (T)

The problem being solved — e.g., classify emails as spam or ham. The task defines what the program must be able to do.

Experience (E)

The training data — examples the system learns from. Each example is called a training instance or sample.

Performance (P)

How well the program does at T — e.g., accuracy (ratio of correctly classified emails). Improves with more experience.

A computer program is said to learn from experience E with respect to some task T and performance measure P, if its performance on T, as measured by P, improves with experience E.

— Tom Mitchell, 1997

Key Insight

Downloading Wikipedia gives a computer more data — but it does not learn anything. That is not ML.

Spam Filter Example (applied to Mitchell's definition)

T — Flag spam email for new messages
E — Training emails labelled spam / ham by users
P — Accuracy (% correctly classified)

Types of ML Systems

Slide 5 / 15

ML systems are classified along three independent axes — these criteria are not mutually exclusive; a real system can combine all of them.

Axis 1 — Supervision

How much human labelling during training?

Supervised

Training data includes desired labels (answers). Used for classification & regression.

Unsupervised

No labels — system discovers structure on its own. Clustering, dim. reduction.

Semisupervised

Mostly unlabelled data + a few labels. Google Photos face grouping.

Reinforcement Learning

Agent acts, gets rewards/penalties, learns an optimal policy. AlphaGo.

Axis 2 — Learning Style

Can the system learn incrementally?

Batch (Offline) Learning

Trained on all available data at once. Can take hours. Then deployed with no further learning. Must retrain from scratch for new data.

Stable Resource-heavy

Online (Incremental) Learning

Trained incrementally in mini-batches. Can learn on the fly. A key parameter is the learning rate: high = fast adapt but forgets; low = stable but slow.

Adaptive Out-of-core capable

Axis 3 — Generalisation

How does it predict on new data?

Instance-Based

Memorises training examples. Generalises by comparing new data to known examples using a similarity measure. Simple example: k-NN.

k-Nearest Neighbours

Model-Based

Builds a model from training examples, then uses the model to make predictions on new data. Most practical ML systems use this approach.

Linear Regression Neural Networks

Example: A state-of-the-art spam filter can be online (incremental), model-based (deep neural network), and supervised — all three axes combined.

Supervised Learning

Slide 6 / 15

Core Idea

The training set you feed to the algorithm includes the desired solutions, called labels. The model learns to map inputs → outputs.

Task 1 — Classification

Predict a discrete class label for each input. The spam filter is a classic binary classification example (spam / ham).

Key concepts:

  • Binary classification: Two classes (yes / no)
  • Multiclass classification: Many classes (digit 0–9)
  • Attributes vs. Features: Attribute = data type; Feature = attribute + value
  • Labels: The "right answers" in training data

Example: Spam Filter

Input: email text & metadata → Output: SPAM or HAM

Task 2 — Regression

Predict a continuous numeric value (target). Requires giving the algorithm many examples including both predictors and the target value.

Key concepts:

  • Predictors: Input features (mileage, age, brand…)
  • Target: Output value to predict (e.g., car price)
  • Note: Some regression algorithms can also classify, and vice versa (e.g., Logistic Regression)

Example: Car Price Prediction

Input: mileage, age, brand → Output: price in USD

Unsupervised Learning

Slide 7 / 15

Core Idea

Training data is unlabelled — the system tries to learn structure without a teacher. The goal is to discover hidden patterns.

Clustering

Group similar instances together — the algorithm finds those connections without being told which group each instance belongs to.

Algorithms:

  • K-Means
  • DBSCAN
  • Hierarchical Cluster Analysis (HCA)

Example: Blog visitors split into groups — 40% male comic fans, 20% young sci-fi readers.

Anomaly Detection

Detect unusual instances — shown mostly normal examples during training, so it learns what "normal" looks like.

Algorithms:

  • One-class SVM
  • Isolation Forest

Also: Novelty detection — find new instances different from all training instances (requires very clean training set).

Dim. Reduction

Simplify data without losing too much information. Merge correlated features into one (feature extraction).

Algorithms:

  • PCA, Kernel PCA
  • LLE, t-SNE

Tip: Reduce dimensions before feeding to supervised ML — faster, less memory, often better performance.

Association Rules

Dig into large datasets and discover interesting relations between attributes automatically.

Algorithms:

  • Apriori
  • Eclat

Example: Supermarket sales reveal: BBQ sauce + chips → steak. Place items together!

Visualisation Algorithms — a special case of unsupervised learning

Feed complex unlabelled data → get a 2D or 3D representation that can be plotted. These algorithms try to preserve the data's structure (e.g., keep separate clusters from overlapping). Helps you understand how data is organised and spot unexpected patterns.

t-SNE is particularly popular: it can reveal semantic clusters even for image datasets (animals vs. vehicles, etc.).

Semisupervised & Reinforcement Learning

Slide 8 / 15

Semisupervised Learning — "A Little Label Goes a Long Way"

Labelling data is time-consuming and costly. In practice you often have many unlabelled instances and few labelled ones. Semisupervised algorithms handle data that is partially labelled.

How it works:

Most algorithms combine unsupervised components (e.g., clustering to group unlabelled data) with supervised fine-tuning on the labelled subset.

Google Photos — a real example

It automatically recognises that the same person A appears in photos 1, 5, 11 (clustering = unsupervised). You add one label per person and it can name everyone across all photos — powerful semisupervised learning.

Deep Belief Networks (DBNs): Built from stacked Restricted Boltzmann Machines (RBMs) trained sequentially in unsupervised fashion, then fine-tuned with supervised techniques — a classic semisupervised architecture.

Reinforcement Learning — "Learn by Doing"

The learning system is called an agent. It observes the environment, selects and performs actions, and gets rewards or penalties in return. It must learn by itself what the best strategy — the policy — is.

The RL Loop

1

Observe the environment state

2

Select action using the current policy

3

Perform the action in the environment

4

Receive reward or penalty

5

Update the policy (learning step)

6

Iterate until an optimal policy is found

DeepMind's AlphaGo

Beat world Go champion Ke Jie (May 2017) by analysing millions of games then playing against itself. Learning was off during championship games — it was applying its learned policy.

Batch vs. Online Learning

Slide 9 / 15

Classification criterion: Can the system learn incrementally from a stream of incoming data?

Batch Learning (Offline Learning)

The system cannot learn incrementally. It must be trained using all available data at once. This typically takes a lot of time and compute resources — done offline.

Once trained and deployed, the system runs without further learning — it just applies what it learned.

To handle new data:

Train a new version on the full dataset (old + new), then swap out the old system. This can be automated — but training from scratch daily is expensive.

Limitations:

  • Takes hours to train on large datasets
  • Requires a lot of CPU, memory, disk, I/O
  • Cannot adapt to rapidly changing data (e.g., stock prices)
  • Impractical for resource-limited devices (e.g., smartphone, Mars rover)

Online Learning (Incremental)

The system trains incrementally by feeding it data individually or in small groups called mini-batches. Each learning step is fast and cheap — learns on the fly.

Great for continuous data streams (stock prices) and for datasets too large to fit in RAM (out-of-core learning).

Key Parameter: Learning Rate

High Learning Rate

Adapts fast to new data but quickly forgets old data.

Low Learning Rate

More inertia. Stable but slower to adapt; less sensitive to outliers.

Challenge: Data Quality

If bad data is fed in (malfunctioning sensor, spam attack on a search engine), performance will decline. Monitor closely and switch learning off if performance drops.

Property Batch Online
New data Retrain from scratch on full dataset Feed incrementally — adapts immediately
Resources High (hours, large RAM/CPU) Low (fast mini-batches, can discard old data)

Instance-Based vs. Model-Based Learning

Slide 10 / 15

How does the system generalise? Most ML tasks require making good predictions on new, unseen instances.

Instance-Based Learning

The system learns examples by heart, then generalises to new cases by comparing them to learned examples using a similarity measure.

Spam Filter Example

A basic instance-based filter flags emails identical to known spam. A smarter version counts shared words — flags an email if it has many words in common with known spam (similarity measure).

k-Nearest Neighbours (k-NN)

Find the k most similar training instances and assign their majority class (or average value) to the new instance. Simple but powerful.

Pros

  • No training phase
  • Naturally adapts as new data added

Cons

  • Slow prediction (compares all instances)
  • Memory scales with training size

Model-Based Learning

Build a model from the training examples, then use that model to make predictions on new data. The model generalises the patterns learned from training.

The Workflow

  1. Study the data — explore & visualise
  2. Select a model — choose algorithm & architecture
  3. Train the model — run algorithm to fit parameters
  4. Apply the model — predict on new instances

Performance Measures

  • Utility / Fitness function: Measures how good the model is (maximise)
  • Cost function: Measures how bad it is (minimise). Linear Regression minimises distance between predictions and training examples.

Pros

  • Fast predictions
  • Compact — only need model params

Cons

  • Requires training phase
  • Must choose model carefully

The true goal of ML is not to perform well on the training data but to make good predictions on new, unseen instances. This is called generalisation.

Model-Based Learning in Practice

Slide 11 / 15

Case study: Does money make people happier? Using GDP per capita to predict life satisfaction — a worked Linear Regression example.

The Model-Based ML Workflow

1Study the data

Download OECD Better Life Index + IMF GDP data. Join, sort by GDP, explore for trends.

2Select a model

Life satisfaction appears to increase linearly with GDP. Select a Linear Model with one predictor (GDP per capita).

3Train the model

Run Linear Regression algorithm — minimises the cost function (distance between predictions and training data). Finds optimal: θ₀ = 4.85, θ₁ = 4.91 × 10⁻⁵.

4Make predictions

Apply the trained model to Cyprus (GDP ~$22,587): predicted life satisfaction ≈ 5.96. Actual: 6.0 — close!

GDP per capita vs. Life Satisfaction

The Linear Model (Equation 1-1)

life_satisfaction = θ₀ + θ₁ × GDP_per_capita

θ₀ = bias (intercept) — 4.85
θ₁ = slope — 4.91 × 10⁻⁵
Model selection: choosing type + architecture

Alternatively, a k-NN regressor on the same data (k=3) also predicts ≈5.77 — this is instance-based, no model parameters needed.

The word "model" can mean: a type of model (Linear Reg.), a specified architecture, or a final trained model — context matters!

Main ML Challenges — Data Problems

Slide 12 / 15

The two main sources of failure in ML: bad algorithms and bad data. For most tasks, data quality matters more than algorithm choice. "Unreasonable Effectiveness of Data."

1. Insufficient Quantity

Even simple ML problems need thousands of examples. Complex problems (images, speech) need millions. A trivial algorithm fed massive data often beats a sophisticated one with little data.

Banko & Brill (2001): NLP algorithms with very different complexity performed similarly once given enough data.

2. Nonrepresentative Data

Training data must represent the cases you want to generalise to. If it's not representative, your model will not generalise — and you may not even know it.

Sampling bias: 1936 Roosevelt election poll — surveyed car/phone owners (wealthy). Predicted Landon wins. Roosevelt won by a landslide.

3. Poor Quality Data

Errors, outliers, and noise in training data — the algorithm struggles to find patterns. Cleaning data is often the most important step.

Options: Fix errors manually, discard outliers, fill in missing values (imputation), or drop the feature entirely.

4. Irrelevant Features

"Garbage in, garbage out." Including irrelevant features degrades model quality. Feature engineering solves this:

Feature selection
Pick the most useful existing features.
Feature extraction
Combine features into more useful ones (dimensionality reduction).
New features
Create new features by gathering more data or augmenting existing.

The Data Quality Checklist

Enough training instances (thousands to millions)
Representative of real-world distribution
Clean — errors and outliers handled
Relevant features selected/engineered
Missing values handled (imputed or dropped)

Main ML Challenges — Algorithm Problems

Slide 13 / 15

Overfitting (High Variance)

The model performs well on training data but fails to generalise to new instances. It has learned the noise, not the signal — too complex relative to the data.

When does it happen?

  • Model has too many parameters relative to training instances
  • Training data has noise / errors
  • Too many unnecessary features

Solutions:

  • Gather more training data
  • Simplify the model (fewer parameters, lower polynomial degree)
  • Apply regularisation — constrain the model's freedom (e.g., reduce θ₁ toward 0). The regularisation hyperparameter controls the tradeoff.
  • Reduce noise in training data (fix errors, remove outliers)

"A model that has learned too much from training data is like a student who memorised all past exam answers but can't solve a new problem."

Underfitting (High Bias)

The model is too simple to capture the underlying structure — poor performance on both training and new data. It makes inaccurate assumptions.

When does it happen?

  • Model has too few parameters
  • Features are not informative enough
  • Too much regularisation (over-constrained)

Solutions:

  • Select a more powerful model with more parameters
  • Provide better features (feature engineering)
  • Reduce regularisation hyperparameter

"A model that underfit is like a student who only memorised one general rule and applies it everywhere — even where it doesn't work."

The Bias-Variance Trade-off

Increasing model complexity reduces bias (underfitting) but increases variance (overfitting risk). The sweet spot is a model complex enough to learn real patterns but simple enough to generalise.

Hyperparameter

A parameter of the learning algorithm (not the model). Set before training - \(\lambda\) (reg. factor)

Testing & Validation

Slide 14 / 15

The Core Problem

How do you know if your model will generalise well to new data before deploying it in production?

Train / Test Split

Split your data into a training set and a test set. Train on the training set, evaluate on the test set to get the generalisation error (out-of-sample error).

Training Set (80%)
Test (20%)

Rule of thumb:

  • Typical split: 80% train / 20% test
  • Low training error + high generalisation error → overfitting
  • High error on both → underfitting
  • The test set should be used only once — to estimate production performance

Data Snooping Bias

If you use the test set too often to tune hyperparameters, you inadvertently overfit to the test set — and the model will likely underperform in production.

Validation Set & Cross-Validation

To tune hyperparameters and select among models without touching the test set, reserve a separate validation (dev) set from training data.

Train (60%)
Val (20%)
Test (20%)

Cross-Validation (k-fold)

Split training data into k folds. Train k times, each time using a different fold as the validation set and the rest for training. Average the scores across all k evaluations.

  • No wasted data — every instance is used for both train and validation
  • More reliable performance estimate (especially for small datasets)
  • k = 5 or 10 is typical

Workflow:

  1. Tune model & hyperparameters using cross-validation on train set
  2. Retrain best model on full training set
  3. Evaluate once on the test set

No Free Lunch Theorem (Wolpert, 1996)

There is no a priori reason to prefer one model over another. The only way to know which model generalises best is to evaluate on data. This requires reasonable assumptions about the data — and that's why testing & validation are not optional, they are fundamental.

End-to-End ML Project Workflow

A professional machine learning project is not just about the model—it's a systematic pipeline. Following these steps ensures reproducibility and avoids critical errors like data snooping bias.

1. Frame the Problem

Define objective. Select performance measure (RMSE, MAE). Check assumptions.

2. Get Data & EDA

Download data. Quick look at structure (head, info, describe). Plot histograms.

3. Split Data

CRITICAL STEP!

Create a Test Set and set it aside. Avoid data snooping bias.

4. Data Prep

Clean data (impute missing). Feature scaling. One-hot encoding. Use Pipeline.

5. Baseline Models

Try many quick-and-dirty models (Linear, SVM, Random Forest) using Cross-Validation.

6. Fine-Tune

Grid Search or Randomized Search for hyperparameters. Analyze best models and errors.

7. Evaluate on Test

Run final model on the hold-out Test Set. Estimate generalization error.

8. Deploy & Monitor

Launch to production. Monitor performance. Retrain regularly.

In skewed datasets (e.g., only 5% of emails are spam), a dumb classifier that always predicts "Not Spam" has 95% accuracy but is useless. We need better metrics.

Key Metrics Formulas

Precision Accuracy of + predictions
TP / (TP + FP)

Use when false positives are costly (e.g., child-safe videos).

Recall Sensitivity / TPR
TP / (TP + FN)

Use when false negatives are critical (e.g., detecting cancer).

F1 Score

Harmonic mean of Precision and Recall. Favors classifiers that have similar precision and recall.

2 × ((Precision × Recall) / (Precision + Recall))
Metrics for Regression Tasks
RMSE (Root Mean Square Error)

Standard metric. Heavily penalizes large errors (outliers).

MAE (Mean Absolute Error)

Preferred if there are many outliers in the dataset.

Precision/Recall Trade-off & ROC

The Decision Threshold

THRESHOLD

Lower Threshold

Captures more positives (↑ Recall), but includes more junk (↓ Precision).

Higher Threshold

Predictions are purer (↑ Precision), but misses some positives (↓ Recall).

Precision-Recall (PR) Curve

When to use: Positive class is rare (imbalanced) or false positives are more important than false negatives.
Metric: Average Precision (AP)

ROC Curve (TPR vs. FPR)

When to use: Classes are balanced or false negatives/positives are equally important.
Metric: Area Under Curve (AUC)

Golden Rule: If the positive class is rare (e.g., fraud detection, disease screening), the ROC curve can be overly optimistic. Always prefer the PR Curve for imbalanced datasets.

Feature Engineering & Search

  • Polynomial Features: Add powers \(x^2, x^3\) to capture non-linear relationships.
  • Interaction Terms: Combine features \(x_1 \times x_2\) to model dependencies.
  • Search Strategies:
    • Grid Search (exhaustive, slow)
    • Randomized Search (faster, explores better)

Regularization: Taming Complexity

Constraining weights to prevent overfitting.

— No Regularization    — With L2 (Ridge)
L2 (Ridge) Minimizes \(\sum w_i^2\). Keeps weights small. Differentiable.
L1 (Lasso) Minimizes \(\sum |w_i|\). Drives weights to zero (feature selection).
Elastic Net Mix of L1 and L2. Good default for sparse features.
Early Stopping Stop training when validation error starts rising.

Gradient Descent & Learning Rate

Gradient Descent Variants

Batch GD

Uses the whole training set at every step. Slow on large data, but stable convergence.

Stochastic GD

Picks one random instance per step. Very fast but erratic (bounces around minimum).

Mini-batch GD

Uses small random sets (e.g., 32). Best balance of speed and stability. GPU friendly.

Learning Rate \(\eta\) Impact

— Too Small (Slow)     — Just Right (Fast & Converges)     — Too Large (Diverges)
Practical Tips for Success
Feature Scaling is Critical: GD converges much faster if all features are on similar scales (e.g., StandardScaler).
Learning Schedules: Start with a large learning rate and reduce it gradually during training.
Monitor Loss: Plot loss curves during training to spot divergence early.

Ensemble Learning Methods

Why Ensembles Work?

"The Wisdom of the Crowd": Aggregating predictions from a group of predictors (even weak ones) often yields better results than the best individual predictor, reducing both bias and variance.

Parallel Ensembles

  • Bagging: Bootstrap Aggregating. Sampling with replacement. Reduces variance.
  • Pasting: Sampling without replacement.
  • Feature Sampling:
    • Random Subspaces (features only)
    • Random Patches (features + instances)

Sequential Ensembles

  • Boosting: Train predictors sequentially, each correcting its predecessor. Reduces bias.
  • AdaBoost: Pay more attention to underfitted training instances.
  • Gradient Boosting: Fit new predictor to the residual errors of the previous one.

Stacking (Stacked Generalization)

Instead of trivial voting, train a Meta-Learner (Blender) to aggregate predictions.

  • Layer 1: Base Predictors
  • Layer 2: Meta-Learner (trained on hold-out set predictions)
  • Can have multiple layers.
Guideline: Boosting is powerful but harder to parallelize than Bagging. Watch out for outliers with AdaBoost.
Guideline: Bagging works best with complex models (low bias, high variance) like Decision Trees.
Guideline: Stacking often wins competitions (Kaggle) but adds complexity to the pipeline.

Random Forests & Feature Importance

How Random Forests Work

An ensemble of Decision Trees, generally trained via the bagging method (typically with max_samples set to the size of the training set).

  • Extra Randomness: Instead of searching for the very best feature when splitting a node, it searches for the best feature among a random subset of features.
  • Result: Greater tree diversity, higher bias, but lower variance → better overall model.

Key Hyperparameters

n_estimators Number of trees in the forest. More is better (but slower).
max_features Max features to consider at each split (critical for diversity).
max_depth Max depth of tree. Limits complexity.
min_samples_leaf Min samples required at a leaf node. Regularizes the tree.
Out-of-Bag (OOB) Evaluation

Since bootstrapping leaves out ~37% of instances for each tree, these "out-of-bag" instances can be used as a validation set without needing a separate hold-out set.

Feature Importance

Sklearn measures importance by looking at how much the tree nodes that use that feature reduce impurity on average.

Useful for feature selection and understanding the data.

Training Deep Neural Networks

The Challenge: Vanishing & Exploding Gradients

Deep networks are hard to train because gradients can get smaller and smaller (vanishing) or larger and larger (exploding) as they propagate back through layers, making lower layers hard to train.

Technique
What it Fixes / Benefit
How to Use (Keras/TF)
Initialization
He Init (for ReLU)
Glorot/Xavier (for Tanh)
Ensures variance of outputs approx. equals variance of inputs. Prevents signal from dying out or exploding at start.
kernel_initializer="he_normal"
Batch Normalization
Stabilization
Fixes "Internal Covariate Shift". Zero-centers and normalizes inputs per layer. Allows higher learning rates.
keras.layers.BatchNormalization()
Optimizers
Adam, RMSProp, Nadam
Adapts learning rate per parameter. Much faster convergence than standard SGD. Avoids local minima better.
optimizer="adam" or "nadam"
Regularization
Dropout, L2, Max-Norm
Prevents overfitting by randomly disabling neurons (Dropout) or penalizing large weights. Forces robust features.
keras.layers.Dropout(rate=0.2)
LR Scheduling
ReduceOnPlateau, Exponential
Start fast, then slow down. Helps fine-tune the final weights when validation loss stops improving.
ReduceLROnPlateau(patience=5)

Key Takeaways & Learning Path

Core Lessons

Workflow First

Success comes from a disciplined end-to-end pipeline: Split data early, clean thoroughly, and baseline before tuning.

Metrics Matter

Don't trust Accuracy blindly. Use Precision/Recall for imbalance and ROC AUC for ranking. Frame the business goal.

Ensembles are Robust

Random Forests and Gradient Boosting are "swiss army knives" for tabular data—often beating complex deep nets.

Deep Learning Stability

Training deep nets requires care: Initialize right, normalize batches, use Adam, and regularize heavily to avoid overfitting.
